Metodologías del desarrollo

Agile

Ventajas Desventajas

  • Tiene una estructura simple pero completa, ya que incluye desde el diseño hasta el lanzamineto.

  • Incluye la retroalimentación para la mejora del proyecto.

  • Se diseña la aplicaión junto al cliente, para evitar mal entendidos con el mismo.

  • Incluye la lógistica del mantenimiento.

  • Si se tiene algún error es la planificaión el diseño, toda la estructura del proyecto se cae.

  • Es una estructura que se puede romper a lo largo de las tomas de decisiónes.

  • A veces el cliente no sabe de forma completa lo que quiere, por lo que gnera errores qué después se cae el proyecto.

  • Una vez qué todo ya está planeado, es difícil hacer cambios sobre la marcha del proyecto

Waterfall (Metodología de Cascada)

Ventajas Desventajas

  • Es fácil de administrar, ya que tiene entregables en todas sus fases de desarrollo.

  • Tanto el proceso correspondiente como los resultados estarán bien documentados.

  • Se puede medir el progreso de forma simple gracias a los entregables.

  • Los clientes no puden cambiar de requisitos constantemente, por lo que el proyecto no se retrasa.

  • No es un modelo ideal para proyectos de gran tamaño.

  • Si no sé los objetivos no son claron será un proyecto poco efectivo.

  • Se dificulta mucho ir atrás en el proceso y hacer cambios.

  • Puede ser difícil para los clientes articular todas sus necesidades al comienzo del proyecto.

XTreme programming (XP - Programación Extrema)

Ventajas Desventajas

  • Permite dar a la programación una extrema organización.

  • Es muy eficiente para la etapa de planificación y pruebas.

  • Gracias a su alta planificación, casi no tiene erroes por lo que es más simple en el momento del desarrollo.

  • El desarrollador casi no se estresa.

  • Es recomendable que solo se use en proyectos de corto plazo o simples.

  • Puede que si algo falla se pierda toda la planificación previa.

  • Es sumamente estricto en el desarrollo de las fases, cosa que puede frustrar.

RAD

Ventajas Desventajas

  • Es un método económico.

  • Los entregables de cada etapa son fáciles de transferir a otra plataforma.

  • Es de fácil inplementación de revisión gracias al fácil traslado.

  • Cuenta con una etapa de abstracción más claro.

  • Es poco utilizado a comparación de otros métodos.

  • No es recomendado para proyectos científicos.

  • Los protopipos nos se pueden escalar por su estructura.

  • Depende de omponentes externos al proyecto.

RUP

Ventajas Desventajas

  • Gracias a su estructura se reducen los riesgos del proyecto.

  • Integra desarrollo con mantenimiento.

  • Es una forma muy eficiente de signar roles.

  • El proceso de software es confiable para satisfacer las necesidades del proyecto.

  • Por sus estrictos roles, es posible que sea muy dificil usarse en proyectos pequeños.

  • Solo se puede utilizar en proyectos muy grandes, ya que tiene una estructura compleja.

DSDM

Ventajas Desventajas

  • La calidad tiende a aumentar ya que el usuario tiene una colaboración directa así como en OS

  • Asegura el desarrollo rápido y ágil.

  • Permite realizar cambios fácilmente

  • Permite reciclar tecnologías de otras aplicaciónes atravez de API's.

  • Puede llegar a ser compleja en cuanto al desarrollo computacional.

  • Cada uno de los integrentes del equipo debe tener poder de tomar decisiones.

  • Tiene fexibilidad en los requerimientos

  • Aveces existen mal entendidos debido a los requerimientos.

Ejemplos de uso:

  • Metodología Agile: Recomendado para pymes que son las suelen tener control de su propio proyecto. Como en una tienda virtual.

  • Metodología Waterfall: No es para proyectos grandes. Esta metodología sería ideal para una página simple, como un folleto en la web.

  • Metodología XTreme programming: Reconmendable para cuando el desarrollador hace algo para si mismo, algo así como lo que Facebook fue al inicio (solo al inicio).

  • Metodología RAD: Para un proyecto simple pero con varios roles, como una página de registro de datos.

  • Metodología RUP: Para una aplicaión muy grande por la definición de roles. Así como una red social.

  • Perfecto para un proyecto con una comunidad activa. Así como el gran proyecto GNU/Linux
